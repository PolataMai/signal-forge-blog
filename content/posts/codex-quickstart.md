## 先把地基摸清楚

一上来就让模型“开始写”通常不是最快路径。高质量执行更像一次侦察：先确认目录结构、已有文件、上层约束和可用工具，再决定改造还是新建。

> 如果当前目录里有 `AGENTS.md`，它几乎等同于项目内的额外系统提示词，必须优先读取。

我通常会先跑这类动作：

```bash
rg --files -g 'AGENTS.md' -g 'package.json' -g 'README.md'
ls -la
find .. -name AGENTS.md -print
```

目标不是搜集所有信息，而是尽快搞清楚三件事：有没有现成代码可复用、有没有上层约束、这次交付的边界是什么。

## 让模型直接执行

当需求已经很明确时，最浪费时间的做法是让模型先写一大段方案，再等你确认，再开始动手。对明确任务，应该让它边做边汇报，而不是停在“建议阶段”。

比较有效的提示方式通常像这样：

```text
目标：做一个可直接部署的个人知识博客
要求：
- 页面炫酷、丝滑、移动端可用
- 能记录 Codex 和 skill 相关知识
- 输出实际代码，不要只给方案
- 最后附带部署说明
```

这种写法的核心是把“交付物”和“约束”提前讲清楚，让模型默认进入执行模式，而不是讨论模式。

## 编辑路径要短

一旦确定方向，接下来要让修改路径尽量短。能直接创建文件，就不要绕去装一堆依赖。能写静态站，就不要先搭一个和内容无关的重型工程。

比如这次博客站，我就直接选了纯静态 HTML、CSS、JavaScript。这样做有三个明显好处：

- 没有 `npm install` 依赖，环境噪声最少。
- 一台任意服务器即可部署，迁移成本低。
- 视觉表现和交互仍然可以做得非常完整。

这类任务里，我更看重“今天就能上线”，而不是“未来可能有无限扩展性”。先把交付物做成，后续再迭代。

## 验证与交付

最后一段常被忽略，但它决定了这次输出是不是真的能落地。至少要覆盖：

- 页面能不能被本地静态服务打开。
- 关键链接是不是都能访问到目标文章。
- 有没有一份清晰的部署说明给未来的自己。

交付时的输出应该像这样有明确闭环：

| 阶段 | 要回答的问题 |
| --- | --- |
| 实现 | 改了什么，用户现在能看到什么 |
| 验证 | 运行了什么检查，哪些还没验证 |
| 部署 | 上线时需要执行哪些命令 |

如果把这三步养成习惯，Codex 就不再只是一个“能回答问题的模型”，而更像一个能协同交付的软件执行器。
